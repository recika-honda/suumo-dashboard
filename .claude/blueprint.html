suumo-dashboard — Blueprint

04_FANGO / FNG26 配下の code project. REINS → forrent.jp 入稿自動化パイプライン

Purpose / Scope

FANGO の不動産仲介業向け AI ツール群のうち、REINS から取得した賃貸物件データを forrent.jp (athome 系入稿チャネル) に自動入稿するパイプライン。最終目標は kento の手作業 0、watch-nyuko の hourly cycle で「広告待ち」を自動消化し、scoring の高い物件は「掲載指示」まで自動昇格させる。

  • In: REINS 検索・抽出、画像分類 (OpenAI Vision)、テキスト生成 (Anthropic + OpenAI)、forrent.jp 入稿フォーム充填、forrent 名寄せスコア取得 + escalation、Notion DB 状態管理、Slack 通知、物確 PF (itandi BB) cascade 画像補完
  • Out: SUUMO / HOMES 等 forrent 以外のポータルへの入稿 (将来候補)、買主側マッチング、契約管理

Key Decisions

  • 2026-05-15 — Phase 3a: 物確 cascade (商号判定なし)。bukaku の detectPlatform(商号) が中小元付業者を網羅できず 16/16 中 1 件しか発火しなかったため、商号マップ式を捨て「物件名 + 部屋番号で物確 PF を順番に試す empirical cascade」へ。itandi 着手済、atbb (要 session 管理) / essquare は次回。
  • 2026-05-15 — Phase 2: REINS 画像不足の早期 exit。raw ≤ 2 (env で上書き可) なら stage 03-06 を skip し、Notion を「画像欠落」へフリップ。Phase 3a cascade とセットで運用 (cascade miss → IMAGE_INSUFFICIENT)。
  • 2026-05-15 — Phase 1: Vision モデル upgrade。gpt-4o-mini+low → gpt-4o+high + プロンプト改訂 (21 を最後の手段、誤分類 hard case 例)。1物件 ~$0.005 → ~$0.05 だが絶対値小で正当化。env で上書き可。
  • 2026-05-15 — REG_FAIL P1 fix。toFullWidth を etcHiyo / etcShohiyo に、sanitizeForForrentText (NFKD + strip Latin combining + NFC recompose + toFullWidth + length cap) を catchCopy / freeComment に。日本語の濁点 U+3099 が NFKD で分解されて HTML entity 展開する罠は NFC recompose で抑止。
  • 2026-05-14 — Phase 7+: forrent escalation 路線。score ≥ 34 で 訂正→掲載→sumapiku+tenpiku ON→再確認→登録 + Slack #ex_fango 通知。閾値 config/score-escalation.json + env SCORE_ESCALATION_THRESHOLD。
  • 2026-05-14 — Phase 7: forrent.js facade 分割。3,136 LOC monolith → 67 LOC facade + 11 サブモジュール。public API 完全互換。
  • 2026-05-13 — ATBB 路線廃案。100件評価で実用マッチ率 65.2% (大手系列流通契約なしが構造要因)。bukaku + Phase 3a cascade に方針集約。
  • 2026-05 — Phase 6: フロント Vanilla 化。Next.js / React / Tailwind / Framer Motion / Socket.IO を全削除。polling 方式で十分。
  • REINS は headed 必須。headless を bot 検出される。NYUKO_HEADLESS=1 のみ headless (debug only)。
  • 不可逆カテゴリ (04 間取り図 / 05 建物外観) は fallback 対象外。Vision 失敗時の pickFallbackCategory で 04/05 を強制充填しない (名寄せスコア破壊 + bukaku 発火条件破壊を防ぐ)。

Non-Functional Requirements

  • パフォーマンス: 1 物件 5-10 分 (Vision 12秒 / forrent fill 1分 / register 1分 が主因)。IMAGE_INSUFFICIENT 早期 exit で raw=0 物件は 25 秒に短縮済。
  • 並列度: batch-nyuko は逐次処理 (REINS / forrent の rate limit 配慮)。並列化は将来課題。
  • 可用性: REINS 運用時間 07:00-23:00 JST。それ以外は login fail (現状 retry なし、watch-nyuko が次サイクルで自動拾い)。
  • セキュリティ: secrets は .env.local のみ、git tracked しない (gitignore 設定済)。Notion トークンは workspace owner 権限。
  • 監視: launchd の com.kento.lamplighter.synctonotion が 1h ごとに Notion sync。失敗は mailbox 経由で Manager Agent。

KPIs

  • 掲載指示昇格率 = SUCCESS かつ escalated:true の件数 / 処理件数。前回バッチ 5/21 = 23.8%。Phase 1+3a で +N 期待。
  • SUCCESS 率 = SUCCESS 件数 / 処理件数。前回 14/21 = 66.7%。P1 fix 後 +3 (REG_FAIL 3 件解消) → 80%+ 期待。
  • cascade hit 率 = Phase 3a cascade が画像取得できた件数 / cascade 発火件数。watch-nyuko 数日で観測。
  • IMAGE_INSUFFICIENT 率 = 画像欠落判定 / 処理件数。素材依頼ワークフロー量の指標。
  • 平均 score = SUCCESS 物件の forrent 名寄せスコア平均。Phase 1 で底上げ期待。
