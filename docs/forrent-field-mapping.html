00入稿前バリデーション (Stage 01 早期 REG_FAIL)

REINS から抽出した時点で必須項目が欠けていれば、forrent にアクセスする前に弾く。設定は config/forrent-required.spec.json に集約 (新項目は JSON に entry 追加で完結)。

REINS field適用範囲 (appliesTo)拒否理由備考
建物名 ALL REINSデータに建物名がありません 全物件種目で forrent サーバ側必須
部屋番号 マンション / アパート REINSデータに部屋番号がありません 戸建・タウンハウス等では required ではない
使用部分面積 マンション / アパート REINSデータに面積（使用部分面積）がありません 元付業者の入力漏れ (例: 100139100018) を 14 秒で reject
判定は trim + falsy 統一。空文字 / 全空白 / null / undefined すべて欠落扱い。詳細追加手順は docs/refactor/adding-required-field.md。

12ほか初期費用 / その他諸費用 / 損保

forrent の費用枠は 2 つに分離:
(A) ほか初期費用 (入会金等) = etcHiyo* — 入居時に一度発生する費用 (鍵交換代・消毒・クリーニング・サポート代・事務手数料)

D. テキスト sanitize ルール

関数用途
norm(str) 全角英数字 → 半角、全角空白 → 半角空白、trim。基本正規化。
toFullWidth(str) 半角カナ + 半角 ASCII (0x20-0x7E) を全角化。biko 専用 (備考欄の半角禁止対応)。dakuten/handakuten 2 文字組合せ ("ｶﾞ"→"ガ") も処理。
sanitizeForLength(text, maxLen) LF/CRLF → 全角スペース置換 → slice(0, maxLen)。文字数制限のある全フィールドで必須 (catchCopy / freeComment / netCatch / netFreeMemo / etcHiyoShosai 等)。
CRLF +1 char anti-pattern (POST 時に \n が \r\n に展開され server side で +1 char 扱い) を予防。

セルフチェックリスト (新規入稿前に kento が確認)

ここに挙げた項目は過去 REG_FAIL になった経験から抽出。新しい物件種目 / エッジケースで失敗したら gotchas.md に追記して将来チェックリストに昇格。
  1. REINS 必須 3 項目: 建物名 / 部屋番号 / 使用部分面積 (マンション・アパートの場合) が全て空でないか
  2. 物件種別: REINS 物件種目が「マンション」かつ 建物構造が木造/軽量鉄骨 → アパート(02) になっているか
  3. 築年月の月精度: 築 12 ヶ月未満が新築判定、それ以上は中古に切替わっているか (年単位だと事故)
  4. 所在地名２の丁目分離: "○○町１丁目" 形式の場合、町村と丁目が正しく分離されたか (azaList は全角数字必須)
  5. 番地: 空でも "-" が入っているか (新築・番地未確定対応)
  6. 号室: 半角 ASCII 10 桁以内か (全角や 11 桁以上はスキップされる)
  7. 周辺施設名: 30 文字超のものが slice されているか (Google Maps 由来の長い施設名で REG_FAIL 多発)
  8. テキスト系: 改行を含む長文が sanitize されているか (CRLF +1 char anti-pattern)
  9. 備考系: 半角カナ・半角英数字が全角化されているか
  10. その他費用合計: 物確値が異常値 (1 円未満 / 50 万円超) でないか
  11. 取引態様 / 管理形態 / 省エネ / BB: 固定値 (4 / 3 / 0 / 1) が確実に入っているか
  12. 会社間流通フラグ: 物件名・部屋番号・詳細住所が全て OFF か (ファンテイズ社方針)
  13. ネット掲載: "保留(3)" になっているか (SUUMO/HOMES 連携前に "掲載(1)" になっていると事故)
  14. 貴社物件コード: fng{reinsId} 形式か
